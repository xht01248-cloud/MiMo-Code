# super-research

[English](README.en.md) · [中文](README.md) · [日本語](README.ja.md) · [Français](README.fr.md) · [Español](README.es.md) · [Русский](README.ru.md)

**Skill автономного исследования** для Claude Code / Claude.ai. Дайте агенту поработать некоторое время (от минут до целой ночи) — и получите **сопоставимые, честные, проверяемые свидетельства**, а не одноразовый ответ из чёрного ящика.

Вдохновлён [autoresearch](https://github.com/karpathy/autoresearch) Карпаты. Skill обобщает его методологию до шести режимов исследования.

---

## Что это

Рабочая дисциплина по принципу «если оставили агента на ночь, утром результату можно верить». Ценность — не в конкретной процедуре, а в **одной и той же дисциплине во всех режимах**:

1. **Контракт до действий** — записать цель, основной артефакт, условие остановки; подтвердить один раз. Дальше — никаких вопросов.
2. **Сначала baseline** — в каждом режиме есть «ответ без моей работы» (немодифицированный код / первые три результата поиска / необработанный датасет). Зафиксировать первым, иначе «лучше» и «значимо» ничего не значат.
3. **Каждый шаг в машиночитаемый журнал, включая провалы** — TSV (не CSV; в описаниях бывают запятые). Тихое отбрасывание провалов — самый быстрый способ обмануть себя и пользователя.
4. **Никаких проверок посреди цикла** — после контракта агент не останавливается спросить «продолжать?». Человек, возможно, спит.
5. **Не жульничать** — не править код eval, не выбрасывать неудобный источник, не заниматься p-хакингом, не отбирать удачные подмножества.

## Шесть режимов

Skill выбирает режим по формулировке пользователя. Можно указать явно.

| Режим | Когда использовать | Триггерные фразы | Подробности |
| --- | --- | --- | --- |
| **Experiment loop** | Улучшать одну систему к числовой цели | "optimize", "tune", "hill-climb", «прогнать эксперименты за ночь» | `references/experiment-loop.md` |
| **Topic survey / 主题调研** | Собрать + синтезировать внешние источники по вопросу | "survey", «обзор литературы», «состояние дел по X» | `references/topic-survey.md` |
| **Quantitative analysis / 量化分析** | Ответить на количественный вопрос по датасету | "analyze this dataset", «предсказывает ли X значение Y» | `references/quant-analysis.md` |
| **Benchmark comparison / 对比评测** | Выбрать одного из N кандидатов | "compare X vs Y", «какую библиотеку выбрать» | `references/benchmark-comparison.md` |
| **Root-cause investigation / 根因排查** | Диагностика регрессии / flaky-теста / просадки производительности | «почему X сломан», «отладить регрессию» | `references/root-cause.md` |
| **Ablation study / 消融实验** | Атрибуция вклада компонентов системы | "ablate", «какие части X важны» | `references/ablation-study.md` |

**Соседние режимы — как отличать** (skill выбирает сам, но можно направить):

- **Experiment loop vs Benchmark** — experiment улучшает *одну* систему; benchmark выбирает из *нескольких*. «Выбрать среди» → benchmark; «сдвинуть эту метрику» → experiment.
- **Experiment loop vs Ablation** — оба редактируют и переизмеряют. Experiment оставляет улучшения; ablation оставляет *каждый* результат и никогда не останавливается досрочно — цель понимание, а не оптимизация.
- **Root-cause vs Experiment loop** — root-cause расследует *сломанный* baseline; experiment карабкается вверх от *рабочего*. Схема журнала и правило остановки различаются.

## Как использовать

### Триггерная фраза

Skill автоматически подгружается на подходящие формулировки. Или задайте явно:

```
Используй режим <mode> из super-research. Работай всю ночь. Артефакты в <dir>/.
```

### Типичный сценарий

```
Вы    : Сравни lib_a, lib_b, lib_c для чистки текста. Benchmark comparison,
        без тюнинга кандидатов. Каждый из 5 кейсов минимум дважды.
Claude: [срабатывает skill → читает SKILL.md + references/benchmark-comparison.md]
        [составляет контракт: кандидаты, матрица, метрики, бюджет справедливости, dir]
        [просит одно подтверждение — ваша последняя возможность вмешаться]
Вы    : Подтверждено
Claude: [автономный цикл — smoke-test harness → прогон матрицы → каждая ячейка
         в matrix.tsv → агрегирование → drop-one-case стабильность → report.md]
        [итоговый отчёт: победитель / ранжирование / стабильность / замечания / логи]
```

Главное: **после контракта Claude больше не задаёт вопросов**. Может работать десятки минут или часы. Можно отойти.

### Артефакты по режимам

| Режим | Рабочая директория | Ключевой журнал | Артефакт |
| --- | --- | --- | --- |
| Experiment loop | `research/<tag>/` (git-ветка) | `results.tsv` | Лучший коммит + отчёт |
| Topic survey | `survey/<tag>/` | `sources.tsv` + `claims.tsv` | Обзор `report.md` с цитированием |
| Quant analysis | `analysis/<tag>/` | `analysis_log.tsv` | `scripts/` + `report.md` + `figures/` |
| Benchmark | `benchmark/<tag>/` | `matrix.tsv` | Ранжирование + рекомендация `report.md` |
| Root-cause | `investigation/<tag>/` | `hypotheses.tsv` + `baseline.md` | Первопричина + доказательство двустороннего обращения |
| Ablation | `ablation/<tag>/` | `ablation.tsv` | Классификация компонентов + отчёт |

Все журналы — **TSV** (табуляция) с заголовком. Директории создаются в текущей рабочей директории; `<tag>` по умолчанию — сегодняшняя дата (напр. `jul7`).

### Форма итогового отчёта

Каждый режим заканчивается компактным markdown (≤ 1 страницы):

- **Contract** — что вы взяли делать
- **Baseline vs final** — числа старта и финиша
- **What worked** — 3–5 пунктов с доказательствами
- **What didn't** — тупики, крэши, противоречия. **Здесь самый ценный сигнал.**
- **Open questions / next steps**
- **Where to look** — журналы, ветка, артефакты

Смысл: **человек должен проверить работу за 5 минут и понять, куда смотреть дальше**. Не сторителлинг.

```
│   ├── quant-analysis.md
│   ├── benchmark-comparison.md
│   ├── root-cause.md
│   └── ablation-study.md
└── evals/                       # для тестирования самого skill
    ├── evals.json               # 8 тест-кейсов с assertions
    ├── toy_repo/                # фикстура experiment-loop (синтетический training-скрипт)
    ├── toy_dataset/             # фикстура quant-analysis (данные Simpson's paradox)
    ├── toy_bench/               # фикстура benchmark (3 cleaner × 5 кейсов)
    ├── toy_regression/          # фикстура root-cause (bisect-совместимый репозиторий из 5 коммитов)
    └── toy_pipeline/            # фикстура ablation (5 переключаемых компонентов)
```

## Progressive disclosure (архитектура)

- **SKILL.md всегда в контексте** — но только общая дисциплина + таблица режимов. Менее 100 строк.
- **`references/<mode>.md` подгружается по запросу** — после выбора режима читается только этот файл.
- **evals/ никогда не попадает в контекст** — используется только при прогоне eval-харнесса skill-creator'а.

Так что при срабатывании skill Claude дочитывает суммарно несколько сотен строк, а не весь свод правил по шести режимам.

## Как тестировать / итерировать сам skill

`evals/evals.json` содержит 8 кейсов, покрывающих все шесть режимов, плюс два теста дисциплины (`handles-crash-gracefully`, `ambiguous-goal-must-clarify`). У каждого кейса:

- `prompt` — инструкция агенту
- `files` — фикстуры, которые нужно положить в рабочую директорию
- `expectations` — программно-проверяемые assertions

Рекомендуемый путь: использовать skill `skill-creator`. Он поднимает по два субагента на кейс (со skill vs baseline), собирает артефакты, прогоняет assertions, формирует `benchmark.json` и HTML-просмотрщик.

Все фикстуры самодостаточны и быстры (`toy_repo/run.py` использует `time.sleep(0.5)`; `toy_regression/setup.sh` каждый раз создаёт свежий репозиторий; `toy_pipeline/pipeline.py` — чисто синтетический scoring).

## Когда его **не** стоит использовать

- Разовые задачи, которые решаются за минуты — обычного диалога достаточно.
- Цели, которые не сводятся к одному числу или одному вопросу с ответом — сначала подумайте.
- Каждый шаг требует одобрения человека — это pair programming, а не автономное исследование.

Ценность — в **объёме × дисциплине**: десять дешёвых экспериментов по одному стандарту сильнее одного остроумного, но непроверенного утверждения. Без дешёвых экспериментов *и* сопоставимого стандарта дисциплина работает вхолостую.
